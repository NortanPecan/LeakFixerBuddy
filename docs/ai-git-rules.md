# Git-конституция для GLM-5

Этот документ содержит правила работы с Git для AI-ассистента (GLM-5). Нарушение этих правил недопустимо без явного письменного разрешения владельца проекта.

---

## 1. Запрещённые операции

GLM-5 **запрещено** выполнять следующие команды без явного текстового согласия владельца:

- `git reset` (любые варианты: `--soft`, `--mixed`, `--hard`)
- `git rebase` (любые варианты)
- `git push --force` или `git push -f`
- `git init` в каталоге, где уже есть папка `.git`
- Удаление папки `.git` (через `rm -rf .git` или аналоги)
- Клонирование поверх текущей директории с существующим `.git`

**По умолчанию** любые откаты делаются через `git revert`.

---

## 2. Обязательные проверки перед работой

Перед началом любого нового этапа GLM-5 обязан:

1. Выполнить `git status` — убедиться, что репозиторий в чистом или ожидаемом состоянии.
2. Выполнить `git remote -v` — убедиться, что существует `origin`, указывающий на GitHub-репозиторий.
3. Если `origin` отсутствует или выглядит некорректно — **остановиться и написать владельцу**, не продолжать работу.

---

## 3. Осмысленные коммиты

- Каждый коммит должен иметь понятное сообщение, описывающее суть изменений.
- Запрещены коммиты с сообщениями типа «fix», «changes», «update» без контекста.
- Формат сообщений: `<type>(<scope>): <description>`
  - `feat(gym): add rest day support`
  - `fix(auth): prevent infinite loading`
  - `docs: update git rules`

---

## 4. Коммит после каждого этапа

После каждого значимого этапа (новый модуль, крупный рефакторинг, изменения в GYM/Skills/Traits/Export):

1. `git status` — проверить изменения
2. `git add .` — добавить изменения
3. `git commit -m "..."`
4. `git push` — отправить на GitHub

---

## 5. Бэкапы перед крупными изменениями

Перед любыми крупными изменениями (миграция БД, перестройка модулей, редизайн):

**Вариант A — ветка:**
```bash
git checkout -b backup/<date>-<topic>
git push -u origin backup/<date>-<topic>
```

**Вариант B — тег:**
```bash
git tag backup-YYYYMMDD-<topic>
git push origin backup-YYYYMMDD-<topic>
```

---

## 6. Обязательное чтение правил

GLM-5 обязан перечитывать этот файл (`docs/ai-git-rules.md`) перед каждым новым этапом:
- Новый модуль
- Крупный рефакторинг
- Изменения в GYM/Skills/Traits/Export

---

## 7. Remote и запрет на локальное «обнуление» репозитория

- Запрещено выполнять `git init` в каталоге, где уже есть папка `.git`.
- Запрещено удалять `.git` (через `rm -rf .git` и любые аналогичные действия), если владелец явно не написал в чате, что понимает последствия удаления всей локальной истории.
- Перед началом любого нового этапа работы GLM-5 обязан проверить:
  - `git status` — репозиторий в чистом или ожидаемом состоянии.
  - `git remote -v` — должен существовать `origin`, указывающий на GitHub-репозиторий.
  - Если `origin` отсутствует или выглядит странно — **остановиться и написать владельцу, ничего не делать дальше**.
- Любые операции, которые могут привести к «новому чистому репозиторию» (повторный `git init`, удаление `.git`, клонирование поверх текущей директории и т.п.), требуют отдельного текстового согласования.

---

## 8. Обязательные push'и и бэкапы

- После каждого значимого этапа (новый модуль, крупный рефакторинг, изменения в GYM/Skills/Traits/Export) GLM-5 обязан сделать:
  - `git status`
  - `git commit` (если есть изменения, с осмысленным сообщением)
  - `git push` на origin.
- Перед любыми крупными изменениями (особенно миграции БД, перестройка модулей, редизайн GYM) GLM-5 обязан:
  - либо создать ветку `backup/<date>-<topic>` и запушить её на origin,
  - либо создать `git tag backup-YYYYMMDD-<topic>` и тоже запушить тег.
- Если по каким-то причинам push невозможен (нет сети и т.п.) — GLM-5 должен явно указать это в `worklog.md` и не выполнять потенциально разрушительные операции до успешного push.

---

## 9. Работа в новых окружениях (песочница/контейнер)

- **GLM-5 никогда не инициализирует git сам.**
- GLM-5 работает только внутри репозитория, который был получен командой `git clone` от владельца.
- Если в текущей директории нет `.git` и нет `origin` — GLM-5 **не считает это ошибкой среды**. Он говорит: «Нужен URL репозитория, чтобы сделать git clone» и ждёт.
- Внутри уже клонированного проекта GLM-5 продолжает следовать всем правилам из этого документа (никаких reset/rebase/git init/rm -rf .git без разрешения, коммит+push после этапов и т.д.).
