# AI Git Rules — Конституция для работы с Git

> **GLM‑5 обязан перечитать этот файл перед каждым новым этапом работы.**

Этот файл содержит правила работы с Git, которые AI‑помощник (GLM‑5) обязан соблюдать без исключений.

---

## 1. Запрет на переписывание истории

**Полный запрет на следующие команды без явного текстового подтверждения от пользователя в чате:**

- `git reset` (любые варианты: `--hard`, `--soft`, `--mixed`)
- `git rebase` (включая `rebase -i`)
- `git push --force` / `git push -f`
- `git push --force-with-lease`
- Любые другие команды, которые переписывают историю коммитов

> **Если пользователь не дал явного подтверждения в текстовом виде — команда запрещена.**

---

## 2. Откат изменений

Если нужно «откатить» изменения:

- **По умолчанию** использовать `git revert` — создаёт новый коммит, отменяющий предыдущий
- `git reset` — **только** с явного подтверждения пользователя
- `git rebase` — **только** с явного подтверждения пользователя

---

## 3. Осмысленные коммиты

- Всегда делать **атомарные, осмысленные коммиты**
- Не работать с большими изменениями «в воздухе» — коммитить по мере завершения логических блоков
- Использовать понятные сообщения коммитов на русском или английском (согласно принятому стилю проекта)
- Формат сообщений: `type(scope): description` (например: `feat(gym): add workout templates`)

---

## 4. Опасные действия — обязательное согласование

**Перед следующими действиями сначала написать пользователю план и ждать подтверждения:**

- Миграции базы данных (изменения `prisma/schema.prisma`, `db:push`, `db:migrate`)
- Массовые удаления файлов или кода
- Большой рефакторинг (затрагивающий 3+ файлов или ключевые модули)
- Git‑операции с историей (rebase, reset, force push)
- Изменения в API, которые могут сломать совместимость
- Удаление или изменение таблиц/полей в БД

---

## 5. Ветки и стабильность

- **`master`/`main`** — стабильная ветка, история не переписывается
- **По умолчанию** крупные изменения делать в отдельной ветке:
  - `feature/название-фичи` — для новых фич
  - `fix/название-фикса` — для багфиксов
  - `refactor/название` — для рефакторинга
- Слияние в `master`/`main` только после проверки и подтверждения

---

## 6. Обязательное чтение перед работой

**Перед каждым новым этапом работы я (GLM‑5) обязан:**

1. Перечитать этот файл
2. Свериться, не нарушаю ли я правила
3. Если планирую опасное действие — сначала спросить у пользователя

**Этапы, требующие перечитывания:**

- Новый модуль (новая фича, новый экран)
- Крупный рефакторинг
- Изменения в GYM / Skills / Traits / Export модулях
- Любые изменения, затрагивающие БД
- Git‑операции, выходящие за рамки `add`, `commit`, `push`, `pull`

---

## 7. Контрольный чек-лист перед каждым коммитом

- [ ] Коммит атомарный (одна логическая задача)
- [ ] Сообщение коммита понятное
- [ ] Нет `git reset` / `rebase` / `force push` без подтверждения
- [ ] Если есть изменения БД — пользователь предупреждён
- [ ] Если есть опасные действия — получено подтверждение
